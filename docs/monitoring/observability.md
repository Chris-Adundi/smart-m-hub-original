# Smart M Hub Observability

## Runtime Signals

Smart M Hub now exposes core runtime signals from the FastAPI backend without changing existing business workflows.

| Signal | Endpoint or Source | Purpose |
|---|---|---|
| Liveness | `/api/health` | Confirms the API process is running. |
| Readiness | `/api/ready` | Confirms MongoDB and the job queue collection are reachable. |
| JSON metrics | `/api/metrics` | Super-Admin-protected counters, route counts, latency windows, and queue depth. |
| Prometheus metrics | `/api/metrics/prometheus` | Super-Admin-protected scrape-friendly metrics for dashboards and alerts. |
| Request logs | stdout/stderr | Structured JSON request logs with request and trace IDs. |
| Frontend errors | `/api/frontend-errors` | Browser route errors from main and super-admin portals. |

## Request Correlation

Every request receives:

- `X-Request-ID`: accepted from the incoming request or generated by the API.
- `X-Trace-ID`: accepted from `traceparent`/`X-Trace-ID` or generated by the API.

These identifiers are written to structured logs and included in API responses, allowing support teams to correlate browser errors, API calls, and worker activity.

## Structured Log Fields

Request logs include:

- `timestamp`
- `level`
- `service`
- `service_version`
- `event_type`
- `request_id`
- `trace_id`
- `method`
- `route`
- `status_code`
- `duration_ms`
- `client_ip`

## Health Check Usage

Use `/api/health` for container liveness checks. It should remain shallow and fast.

Use `/api/ready` for load balancers and deployment gates. A failed readiness response means the instance should not receive production traffic.

```mermaid
flowchart LR
    LB[Load Balancer] --> Ready[/api/ready]
    Kube[Kubernetes or Process Manager] --> Health[/api/health]
    API[FastAPI Instance] --> Mongo[(MongoDB)]
    API --> Jobs[(jobs collection)]
```

## Metrics Notes

The current registry is intentionally lightweight and in-process. In multi-instance production, Prometheus should scrape every instance and aggregate metrics centrally.

Metrics endpoints require authenticated Super Admin access. Production monitoring should use a dedicated internal credential or an internal network path that preserves this control.

Recommended next production integration:

- Prometheus for metrics collection.
- Grafana for dashboards.
- OpenTelemetry Collector when separate workers and services are deployed independently.
